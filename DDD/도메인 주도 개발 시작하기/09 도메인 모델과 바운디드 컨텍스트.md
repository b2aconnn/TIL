# 도메인 모델과 경계

**처음 도메인 모델을 만들 때 빠지기 쉬운 함정이 도메인을 완벽하게 표현하는 단일 모델을 만드는 시도를 하는 것이다.**

- 한 도메인은 여러 하위 도메인으로 구분되기 때문에 한 개의 모델로 여러 하위 도메인을 모두 표현하려고 시도하면 오히려 모든 하위 도메인에 맞지 않는 모델을 만들게 된다.

<br>

👉 **논리적으로 같은 존재처럼 보이지만 하위 도메인에 따라 다른 용어를 사용하는 경우도 있다.**

**상품을 의미하는 각 하위 도메인**

- 카탈로그 도메인의 상품 : Product
- 검색 도메인에서의 상품 : Document

**회원을 의미하는 각 하위 도메인**

- 회원 도메인의 회원 : Member
- 주문 도메인의 회원 : Orderer
- 배송 도메인의 회원 : Sender

<br>

💁‍♂️ **이처럼 하위 도메인마다 같은 용어라도 의미가 다르고 같은 대상이라도 지칭하는 용어가 다를 수 있기 때문에 한 개의 모델로 모든 하위 도메인을 표현하려는 시도는 올바른 방법이 아니다.**

- 하위 도메인마다 사용하는 용어가 다르기 때문에 하위 도메인마다 모델을 만들어야 한다.
- 각 모델은 명시적으로 구분되는 경계를 가져서 섞이지 않도록 해야한다.
  - 모델이 섞이기 시작하면 모델의 의미가 약해진다.
  - 여러 도메인의 모델이 서로 얽히면 각 하위 도메인마다 다르게 발전하는 요구사항을 모델에 반영하기 어려워진다.

<br>

💁‍♂️ **구분되는 경계를 갖는 컨텍스트를 DDD에서는 바운디드 컨텍스트(Bounded Context)라고 부른다.**

- 모델은 특정한 컨텍스트 하에서 완전한 의미를 갖는다.
- 같은 Product라도 카탈로그 컨텍스트와 재고 컨텍스트에서 의미가 서로 다르다.

<br>

# 바운디드 컨텍스트

**바운디드 컨텍스트는 모델의 경계를 결정하며 한 개의 바운디드 컨텍스트는 논리적으로 한 개의 모델을 갖는다.**

- 바운디드 컨텍스트는 용어를 기준으로 구분한다.
- 바운디드 컨텍스트는 실제로 사용자에게 기능을 제공하는 물리적 시스템으로 도메인 모델은 이 바운디드 컨텍스트 안에서 도메인을 구현한다.

<br>

👉 **하위 도메인에 두 개 이상의 바운디드 컨텍스트 구현**

- 주문 하위 도메인의 경우 주문을 처리하는 팀과 복잡한 결제 금액 계산 로직을 구현하는 팀이 따로 있을 경우, 주문 바운디드 컨텍스트와 결제 금액 계산 바운디드 컨텍스트가 존재할 수 있다.

<img width="350" alt="스크린샷 2023-08-08 오후 10 59 19" src="https://github.com/b2aconnn/TIL/assets/101120568/4529f393-b379-49b9-8d73-ae058bf5b2a8">

<br>

👉 **두 하위 도메인을 하나의 바운디드 컨텍스트에서 구현**

- 카탈로그와 재고 관리가 아직 명확하게 구분되지 않은 경우 두 하위 도메인을 하나의 바운디드 컨텍스트에서 구현할수도 있다.

<img width="230" alt="스크린샷 2023-08-08 오후 11 03 24" src="https://github.com/b2aconnn/TIL/assets/101120568/22c6ab3e-621b-4ae5-b78e-ee250af91a6e">

<br>

👉 **여러 하위 도메인을 한 개의 바운디드 컨텍스트에서 구현**

- 차후에 하위 도메인별로 기능을 확장할 수 있기 때문에 하위 도메인의 모델이 섞이지 않도록 주의해야 한다.
  - 모델이 섞이면 확장하기 어렵게 되어 서비스 경쟁력을 떨어뜨리는 원인이 될 수 있다.
- 물리적인 바운디드 컨텍스트가 한 개이더라도 내부적으로 패키지를 활용해서 논리적으로 바운디드 컨텍스트를 만든다.

<img width="400" alt="스크린샷 2023-08-08 오후 11 12 12" src="https://github.com/b2aconnn/TIL/assets/101120568/f795ca4d-25e7-4d7f-a8d3-94f78fcb63b6">

<br>

👉 **바운디드 컨텍스트는 구현하는 하위 도메인에 알맞는 모델을 포함한다**

- 같은 사용자라 하더라도 주문 바운디드 컨텍스트에서는 Orderer의 밸류가 되고, 회원 바운디드 컨텍스트에서는 Member인 애그리거트 루트가 된다.
- 카탈로그의 Product는 상품이 속할 Category와 연관을 갖지만, 재고의 Product는 카탈로그의 Category와 연관을 맺지 않는다.

<br>

# 바운디드 컨텍스트 구현

💁‍♂️ **바운디드 컨텍스트는 도메인 모델 뿐만 아니라 도메인 기능을 사용자에게 제공하는데 필요한 표현 영역, 응용 서비스, 인프라스트럭처 영역, DB 테이블 스키마 모두를 포함한다.**

<img width="200" alt="스크린샷 2023-08-08 오후 11 29 04" src="https://github.com/b2aconnn/TIL/assets/101120568/c693049f-bced-4eb8-901f-564f204194c6">

<br>

👉 **모든 바운디드 컨텍스트는 반드시 도메인 주도로 개발할 필요는 없다.**

- 복잡한 도메인 로직을 갖지 않는다면 DAO와 데이터 중심의 밸류 객체를 이용해 CRUD 방식으로 구현해도 된다.

<img width="380" alt="스크린샷 2023-08-08 오후 11 33 09" src="https://github.com/b2aconnn/TIL/assets/101120568/52da33a3-41d9-4f15-8a2e-c0b49ee0b9ff">

<br>

👉 **한 바운디드 컨텍스트에 도메인 주도적 개발, 데이터 중심 개발을 혼합해서 사용**

- CQRS 패턴을 활용해 상태 변경하는 명령 기능과 조회하는 쿼리 기능을 위한 모델을 구분하는 패턴으로 구현할 수 있다.

<img width="350" alt="스크린샷 2023-08-08 오후 11 38 20" src="https://github.com/b2aconnn/TIL/assets/101120568/81911cef-e34e-4d84-9201-21c5cd5a54a8">

<br>

👉 **각 바운디드 컨텍스트는 서로 다른 구현 기술을 사용할 수 있다.**

- 스프링 MVC와 JPA/하이버네이트를 사용하는 바운디드 컨텍스트가 있을 수 있고, Netty를 이용한 REST API와 MyBatis를 사용하는 바운디드 컨텍스트가 존재할 수 있다.

<br>

# 바운디드 컨텍스트 간 통합

카탈로그 하위 도메인에 추가로 추천 바운디드 컨텍스트가 필요하다면 기존 카탈로그 바운디드 컨텍스트와 통합이 필요하게 된다.

<p align="center">
  <img width="474" alt="스크린샷 2023-08-10 오후 11 45 19" src="https://github.com/b2aconnn/TIL/assets/101120568/6587bcca-2e0d-44cd-a2bf-3bb930db9f69">
</p>

**카탈로그와 추천 바운디드 컨텍스트 간 통합이 필요한 기능**

- 사용자가 제품 상세 페이지를 볼 때, 보고 있는 상품과 유사한 상품 목록을 하단에 보여준다.
